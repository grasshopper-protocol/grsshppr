# ADR-0004: Timezone handling — calendar dates and slot wire format

- **Status:** Accepted
- **Date:** 2026-06-23
- **Discipline:** Engineering
- **Source RFC:** none — direct (forced by a production booking bug, see
  [post-mortems/2026-06-23-booking-timezone.md](../postmortems/2026-06-23-booking-timezone.md))

## Context

Booking spans three clocks: the mentor's timezone (where availability is
defined), the mentee's timezone (where slots are displayed), and UTC (where
instants are stored and validated). A bug surfaced where availability set for
Monday showed slots on Sunday, and booking failed with a generic error.

Two root causes:

1. The client derived the week-start date string with `Date.toISOString()`. In
   positive-UTC zones (e.g. SGT, UTC+8) local midnight Monday is still Sunday in
   UTC, so the date rolled back a day.
2. Generated slots were emitted as naive `YYYY-MM-DDTHH:mm:ss` with no offset.
   The booking API validates with `z.string().datetime()`, which requires a
   `Z`/offset — so every booking POST 400'd.

Without an explicit rule, any new date/slot code can reintroduce either fault.

## Options

- **Store and pass naive local strings everywhere** — simple, but ambiguous
  across zones and exactly what caused the bug. Rejected.
- **Use a date library (Luxon/date-fns-tz) for all conversions** — robust, but a
  new dependency for a problem solvable with a small, tested helper. Deferred.
- **Adopt two narrow rules + a local offset-correction helper** — no new
  dependency, fixes both faults, easy to test. **Chosen.**

## Decision

1. **Never derive a calendar date from `toISOString()`.** Build `YYYY-MM-DD`
   from local date parts (`getFullYear`/`getMonth`/`getDate`) via a `toDateStr()`
   helper.
2. **Slot wire format is always offset-aware ISO** (a real UTC instant).
   Availability slots are generated by converting the mentor's local time to a
   true UTC instant honoring the mentor's timezone (`zonedTimeToUtc()`
   offset-correction, DST-tested). The client groups slots by *local* date for
   display.

## Consequences

- Booking is correct across zones and DST; the API's `z.string().datetime()`
  validation passes because the wire format carries an offset.
- All new date/slot code must follow these two rules — calendar strings from
  local parts, instants as offset-aware ISO.
- A future move to a tz library (if conversions multiply) would supersede this
  ADR, not contradict it.
